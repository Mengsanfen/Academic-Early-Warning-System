## 1. 项目概述

### 1.1 项目背景

在高校学生管理工作中，学业预警是一项重要的工作内容。传统的学业预警主要依赖人工统计和经验判断，存在以下问题：

- **数据分散**：学生成绩、出勤、行为等数据分散在不同系统中，难以综合分析
- **预警滞后**：人工统计周期长，往往在问题严重时才发现
- **标准不一**：预警标准依赖个人经验，缺乏统一规范
- **跟进困难**：预警后的处理过程缺乏有效跟踪

### 1.2 项目目标

构建一个基于多源数据融合和可配置规则引擎的学业预警系统，实现：

1. **多源数据整合**：统一管理成绩、出勤等多种学生数据
2. **智能预警识别**：通过规则引擎自动识别学业风险学生
3. **分级预警管理**：支持不同等级的预警（一般、严重、紧急）
4. **闭环处理流程**：预警生成→通知→处理→反馈的完整闭环
5. **可视化展示**：通过图表直观展示预警统计和趋势

### 1.3 系统角色

| 角色 | 权限说明 |
|------|----------|
| **管理员** | 系统配置、规则管理、数据导入、统计查看 |
| **辅导员** | 查看预警学生、处理预警、记录跟进情况 |
| **学生** | 查看个人预警状态、改进建议 |

### 1.4 预警类型

| 预警类型 | 说明 | 数据来源 |
|----------|------|----------|
| **成绩预警** | 挂科、GPA下降、学分不足 | 成绩数据 |
| **出勤预警** | 旷课过多、长期缺勤 | 考勤数据 |
| **毕业风险预警** | 综合评估毕业风险 | 多源数据融合 |

### 2.1 技术选型

### 2.3 设计理念

1. **前后端分离**：职责清晰，便于并行开发和维护
2. **模块化设计**：各模块低耦合，便于扩展
3. **配置化规则**：预警规则可配置，无需修改代码
4. **异步处理**：预警扫描等耗时操作异步执行

## 5. 规则引擎设计

### 5.1 规则引擎概述

规则引擎是本系统的**核心组件**，负责将预定义的规则应用到学生数据上，自动识别存在学业风险的学生。采用"解析器 + 执行器"的设计模式，支持规则的可配置化和动态加载。

### 5.2 架构设计

```
┌─────────────────────────────────────────────────────────────────┐
│                      规则引擎架构                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐        │
│  │  规则定义   │───▶│  规则解析   │───▶│  规则执行   │        │
│  │  (JSON)    │    │  (Parser)   │    │  (Executor) │        │
│  └─────────────┘    └─────────────┘    └─────────────┘        │
│         │                  │                  │                │
│         ▼                  ▼                  ▼                │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐        │
│  │ 存储在MySQL │    │ 转为Python  │    │ 匹配学生数据│        │
│  │ rules表    │    │ 可执行对象  │    │ 生成预警    │        │
│  └─────────────┘    └─────────────┘    └─────────────┘        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.3 规则定义格式

规则以 JSON 格式存储在数据库的 `rules` 表中，格式如下：

```json
{
  "rule_code": "SCORE_FAIL_2",
  "name": "连续挂科预警",
  "type": "score",
  "conditions": {
    "metric": "score",
    "operator": "<",
    "threshold": 60,
    "aggregation": "count",
    "time_window": "semester",
    "min_count": 2
  },
  "level": "warning",
  "message_template": "本学期已有 {count} 门课程不及格，请注意学业"
}
```

### 5.4 支持的规则类型

#### 5.4.1 成绩类规则

| 规则编码 | 规则名称 | 条件说明 | 预警级别 |
|----------|----------|----------|----------|
| SCORE_FAIL_2 | 连续挂科预警 | 单学期挂科≥2门 | warning |
| SCORE_FAIL_3 | 严重挂科预警 | 单学期挂科≥3门 | serious |
| SCORE_GPA_LOW | GPA过低预警 | 累计GPA < 2.0 | warning |
| SCORE_GPA_DANGER | GPA危险预警 | 累计GPA < 1.5 | urgent |

**条件示例（JSON）：**
```json
{
  "metric": "score",
  "operator": "<",
  "threshold": 60,
  "aggregation": "count",
  "time_window": "semester",
  "min_count": 2
}
```

#### 5.4.2 出勤类规则

| 规则编码 | 规则名称 | 条件说明 | 预警级别 |
|----------|----------|----------|----------|
| ATTEND_ABSENT_RATE | 缺勤率预警 | 缺勤率 > 30% | warning |
| ATTEND_CONTINUOUS | 连续缺勤预警 | 连续缺勤 ≥ 3次 | serious |
| ATTEND_LONG_TERM | 长期缺勤预警 | 累计缺勤 ≥ 10次 | urgent |

**条件示例（JSON）：**
```json
{
  "metric": "absence_rate",
  "operator": ">",
  "threshold": 0.3,
  "time_window": "semester"
}
```

#### 5.4.3 毕业风险类规则

| 规则编码 | 规则名称 | 条件说明 | 预警级别 |
|----------|----------|----------|----------|
| GRAD_CREDIT_LOW | 学分不足预警 | 已修学分/要求学分 < 80% | warning |
| GRAD_CREDIT_DANGER | 学分危险预警 | 已修学分/要求学分 < 60% | urgent |
| GRAD_COMPOSITE | 综合毕业风险 | 多条件综合评估 | serious |

**条件示例（JSON）：**
```json
{
  "metric": "credit_ratio",
  "operator": "<",
  "threshold": 0.8,
  "expected_year": 4,
  "current_year": 3
}
```

### 6.6 成绩生成算法

```python
# scores.py

def generate_score(student_type: str) -> float:
    """
    根据学生类型生成成绩

    Args:
        student_type: excellent/normal/edge/risk

    Returns:
        成绩分数
    """
    ranges = {
        "excellent": (85, 100),
        "normal": (60, 85),
        "edge": (50, 70),
        "risk": (30, 55),
    }

    min_score, max_score = ranges[student_type]
    # 使用正态分布，使成绩更自然
    score = random.gauss((min_score + max_score) / 2, 8)
    return max(min_score, min(max_score, score))
```

### 9.4 Nginx 配置

```nginx
# nginx.conf
events {
    worker_connections 1024;
}

http {
    include       /etc/nginx/mime.types;
    default_type  application/octet-stream;

    # Gzip 压缩
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml;

    # 后端 API 代理
    upstream backend {
        server backend:8000;
    }

    server {
        listen 80;
        server_name localhost;

        # 前端静态文件
        location / {
            root /usr/share/nginx/html;
            index index.html;
            try_files $uri $uri/ /index.html;
        }

        # API 代理
        location /api {
            proxy_pass http://backend;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        }

        # API 文档
        location /docs {
            proxy_pass http://backend;
        }
    }
}
```

## 11. 2026-04-13 功能增强：规则实施对象与课程类型

### 11.1 业务目标

为贴近高校真实业务场景，系统在本轮迭代中补充了两类核心能力：

1. 规则实施对象控制：每条预警规则可限定作用范围，支持面向全体学生、指定年级、指定班级执行，避免“一套规则覆盖全校所有年级”的业务失真。
2. 课程类型分类：课程管理新增 `必修课 / 选修课 / 公共课 / 专业课 / 实践课` 五类属性，规则执行时可按课程类型区分学分和成绩要求，用于模拟高校培养方案中的差异化标准。

### 11.2 数据模型扩展

#### 11.2.1 规则表 `rules`

新增字段：

- `target_type`：规则实施对象类型，枚举值为：
  - `all`：全体学生
  - `grades`：按年级实施
  - `classes`：按班级实施
- `target_grades`：JSON 数组，保存实施年级列表，例如 `['2024', '2025']`
- `target_classes`：JSON 数组，保存实施班级 ID 列表，例如 `[1, 3, 5]`

字段设计原则：

- `target_type = all` 时，`target_grades` 与 `target_classes` 统一清空。
- `target_type = grades` 时，仅允许 `target_grades` 生效。
- `target_type = classes` 时，仅允许 `target_classes` 生效。

#### 11.2.2 课程表 `courses`

新增字段：

- `course_type`：课程类型，枚举值包括：
  - `required`：必修课
  - `elective`：选修课
  - `public`：公共课
  - `professional`：专业课
  - `practice`：实践课

设计目的：

- 为学分不足、挂科门数、低绩点等规则提供课程类型维度。
- 支持高校中“必修学分要求高于选修学分要求”的真实管理场景。

### 11.3 运行时兼容与数据库同步

为兼容已部署数据库，`backend/app/schema_sync.py` 增加了运行时自动补齐逻辑：

- 自动检查并补充 `rules.target_type`
- 自动检查并补充 `rules.target_grades`
- 自动检查并补充 `rules.target_classes`
- 自动检查并补充 `courses.course_type`
- 对历史课程数据统一回填 `course_type = required`

这意味着旧数据库在不手工迁移的情况下，也能在服务启动后自动获得本轮新增字段。

### 11.5 规则执行链路调整

文件：`backend/app/core/rule_engine/simple_engine.py`

规则执行过程改造如下：

1. 根据 `Rule.target_type` 确定目标学生集。
2. 当 `target_type = grades` 时，仅对选定年级学生执行。
3. 当 `target_type = classes` 时，仅对选定班级学生执行。
4. 当规则条件中包含 `course_type` 时，成绩指标和学分指标只统计该课程类型的数据。
5. 支持的课程类型敏感指标包括：
   - `score`
   - `avg_score`
   - `fail_count`
   - `gpa`
   - `earned_credit`
   - `failed_credit`
   - `attendance_rate`
   - `absence_count`
   - `late_count`

典型业务示例：

- 必修学分不足：`metric = earned_credit`，`course_type = required`
- 选修学分不足：`metric = earned_credit`，`course_type = elective`
- 必修挂科门数偏多：`metric = fail_count`，`course_type = required`

### 11.9 枚举兼容修复（登录首页 500 修复）

问题现象：

- 用户登录后访问 `GET /api/v1/dashboard/overview` 返回 500。
- 根因是 `rules.target_type` 与 `courses.course_type` 在数据库中保存为小写值（如 `all`、`required`），而 ORM 枚举按默认大写枚举名解析，导致查询包含规则或课程关联时触发 `LookupError`。

修复方案：

- 在 `backend/app/models/rule.py` 中将 `target_type` 改为按枚举 `value` 持久化与反序列化。
- 在 `backend/app/models/course.py` 中将 `course_type` 改为按枚举 `value` 持久化与反序列化。
- 补充回归测试，验证数据库中存在小写枚举值时，规则列表、课程列表和仪表盘概览均可正常读取。

验证结果：

- 本地真实数据库复测 `dashboard/overview` 已恢复正常。
- 回归测试 `pytest backend/tests/test_rule_course_scope.py -q` 结果为 `7 passed`。

### 11.10 规则静态子路由优先级修复

问题现象：

- 访问 `GET /api/v1/rules/templates`、`GET /api/v1/rules/target-options` 返回 422。
- 前端规则配置页因此无法加载指标、目标范围和课程类型元数据。

根因：

- `rules` 路由中 `GET /api/v1/rules/{rule_id}` 注册顺序早于 `GET /templates`、`GET /grades`、`GET /target-options`。
- FastAPI/Starlette 按注册顺序匹配路由，导致 `templates`、`target-options` 被当作 `rule_id` 路径参数解析，进而触发整数校验失败并返回 422。

修复方案：

- 将 `GET /templates`、`GET /grades`、`GET /target-options` 等静态子路由整体前移。
- 将 `GET /{rule_id}`、`PUT /{rule_id}`、`DELETE /{rule_id}`、`POST /{rule_id}/toggle` 等动态路由后置。
- 新增回归测试，锁定静态规则子路由必须优先于动态详情路由。

验证结果：

- 规则页元数据接口已恢复正常。
- 回归测试 `pytest backend/tests/test_rule_course_scope.py -q` 结果为 `8 passed`。
